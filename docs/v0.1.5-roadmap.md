# v0.1.5+ roadmap — ideas inspired by clash-verge-rev

This document is a punch list of layout and feature ideas for Cool Tunnel
that came out of researching [clash-verge-rev][cv]. Nothing here is
committed scope — pick what you want for v0.1.5 and slice the rest into
later releases.

[cv]: https://github.com/clash-verge-rev/clash-verge-rev

Today's Cool Tunnel layout (for reference):

```
┌─ HeaderView (status / firewall) ─────────────┐
│  ControlPanelView (start/stop/mode)          │
│  ConnectionFormView (profile picker)         │
│  LogConsoleView (live engine logs)           │
└─ SettingsView (modal sheet) ─────────────────┘
```

Clash Verge Rev's layout instead is a persistent sidebar with **8 pages**:
Home / Proxies / Profiles / Connections / Rules / Logs / Unlock / Settings,
where Home is a dashboard of toggleable cards (proxy, traffic, profile,
mode, TUN, IP/Geo, Clash info, system info, latency).

We are **not** going to copy that wholesale — Cool Tunnel wraps a single
upstream binary with no rules engine, no kernel mode switching, no
subscription system. The ideas below are scoped to what naive + a Rust
core can actually deliver.

## Top 5 candidates (recommended order)

### 1. Menu-bar (NSStatusItem) item with quick toggles  *— small, ship first*

**Why first**: highest user-value-per-line-of-code. People run a proxy
app to *not* think about it; the menu bar is where it should live.

**Scope**:
- `NSStatusItem` with the Cool Tunnel icon
- Menu items: Start / Stop, profile picker submenu, "Show window…",
  "Quit"
- Show current state in the icon (e.g. dot colour or filled vs outline)
- Optional: badge with active mode

**Where**: app-level (`COOL-TUNNEL/App/`); no changes to the existing
panes. Reuses the `TunnelOrchestrator` that's already an `@Observable`.

**Difficulty**: small. ~150 lines of AppKit + SwiftUI bridge. No Rust
changes.

**Risk**: AppKit + SwiftUI menu bar items have some lifecycle quirks
(retain cycles around the orchestrator) but they're well-trodden.

---

### 2. Latency / IP-Geo card on the Header  *— small*

**Why**: the current header just says "running" / "stopped". Users want
to know the proxy is *actually working* — what IP they look like, how
fast it is.

**Scope**:
- New `HeaderView` row: ping to a test URL through the tunnel (e.g.
  `https://www.google.com/generate_204`) + extracted IP from
  `https://ipinfo.io/json` + country code
- 30-second auto-refresh while tunnel is up
- A "Test now" button to force-refresh

**Where**: extends `HeaderView` (one section). Reuses the existing
`runLatencyTest` machinery in the Rust core.

**Difficulty**: small. The Rust side already has `run_diagnostics` and
`run_latency` — we just call them on a timer and surface the result.

**Risk**: low. Make sure the auto-refresh debounces if the user rapidly
toggles start/stop (we have `Debouncer` available).

---

### 3. Settings reorganised into a sidebar with sections  *— small-medium*

**Why**: the modal is already groaning under five sections (Direct
Domains, This Mac, Naive Binary, Behaviour, plus the Test/Picker
controls). It will only get worse as we add more.

**Scope** — split into a `NavigationSplitView` with these sections:

| Section | Contents |
| --- | --- |
| General | Skip confirmations, start at login |
| Routing | Direct domains list, smart-mode toggle |
| Engine | Naive binary path + Test button + chip detection |
| Appearance | Light / Dark / System, accent colour, font size |
| Advanced | Open Application Support folder, export diagnostics, version + commit hash |

**Where**: rewrites `SettingsView.swift` from `Form` to
`NavigationSplitView`. The current `naiveBinarySummary` and
`chipDetectionRow` slot into Engine; the domain list into Routing.

**Difficulty**: small-medium. ~400 lines of SwiftUI; mostly mechanical.

**Risk**: low. No new behaviour — purely UX reorganisation. Worth
keeping the modal sheet *as well* for "settings cog" entry points so
the experience is consistent across windows.

---

### 4. Live Connections pane  *— medium-large*

**Why**: clash-verge-rev's killer feature. A live table of active flows
(host, bytes ↑↓, duration, originating process) is the single most
useful diagnostic when a site won't load through the proxy.

**Scope**:
- Promote `LogConsoleView` into a tabbed "Activity" pane:
  - **Logs** — current live log stream
  - **Connections** — table of active flows
  - **Stats** — bytes/sec graph (uses existing `metrics` module)
- Rust core needs a new `monitor::connections()` function that parses
  `lsof -nP -p <pid> -iTCP -F` output into structured `Connection`
  rows (host, bytes ↑↓ from netstat, originating PID via `lsof`)
- Stream connection deltas as a new `Event::ConnectionsSnapshot` over
  the JSON-over-stdio channel; the Swift table updates incrementally
- 1-second refresh, debounced

**Where**: replaces `LogConsoleView` with a `TabView`. Adds
`core/src/monitor/connections.rs`. Adds a new event variant in
`core/src/protocol.rs`.

**Difficulty**: medium-large. Naive doesn't expose per-flow accounting
natively — this requires shelling out to `lsof` + `netstat`/`nettop`
and reconciling the two. Per-process attribution needs `lsof -nP`
which is privileged-ish; might require a helper tool for full data.

**Risk**: medium. Per-process info on macOS is increasingly restricted
(System Integrity Protection, app sandbox); we may only be able to
get per-port info reliably without a privileged helper. Start with the
flow-level view (no process attribution) and add process info later.

---

### 5. Profile cards with subscription URL import + auto-refresh  *— medium*

**Why**: today's `ConnectionFormView` is a single picker. Users with
multiple servers (work / home / different countries) want to manage
them like cards.

**Scope**:
- Convert profile picker into a `LazyVGrid` of cards:
  name, server endpoint, last updated, bytes since start, "Use" button
- "Import from URL" button: paste a subscription URL → Rust core
  fetches → parses (naive doesn't have a subscription format yet, so
  define a simple JSON schema) → adds profiles
- Per-profile auto-refresh interval (Settings → Engine)
- Drag-to-reorder
- Drag-and-drop a `.json` profile file onto the window to import

**Where**: rewrites `ConnectionFormView`. Adds
`core/src/config/subscription.rs` with a fetch-and-parse function.

**Difficulty**: medium. SwiftUI side is mostly mechanical; the
Rust-side subscription fetcher needs HTTPS support (add `reqwest` or
just shell out to `curl --proxy` via the existing helper).

**Risk**: medium. `reqwest` is a heavy dependency for one feature.
Consider shelling out to `curl` via `tokio::process` instead — keeps
the Rust core lean.

---

## Honourable mentions (v0.2.0+)

- **URL scheme handler** (`cooltunnel://import?url=...`) for one-click
  subscription install. Small AppKit work.
- **Hotkeys** (start/stop, mode switch). Tricky on macOS without
  Accessibility permissions — clash-verge-rev does it with `tauri-plugin-global-shortcut`; we'd use `MASShortcut` or a SwiftUI equivalent.
- **Lightweight mode** — close the GUI but keep the proxy running.
  Effectively "hide window, stay alive in tray". Requires the menu-bar
  item from #1.
- **Custom CSS / accent colour theming**. Settings → Appearance.
- **WebDAV backup** of profiles. Out-of-scope unless users ask.
- **Export diagnostic info** (Settings → Advanced → Save zip). Small;
  just bundles the support directory + recent logs.
- **Per-mode tray icon** (different shade for Direct / Smart / Global).

## What we explicitly didn't include

These are clash-verge-rev features that don't map onto Cool Tunnel:

- **Rule editor** — naive doesn't have a rules engine; routing is the
  smart-PAC + direct-domains list we already have
- **Proxy groups + node latency tests across providers** — naive is a
  single upstream
- **TUN mode + privileged service** — way out of scope; would require
  an installer + helper tool
- **Kernel switcher** (mihomo / clash) — we are bound to naive
- **Web UI launcher / external controller** — naive has no API
  surface to control externally
- **GeoData updater** — no GeoIP rules, so nothing to update

## Suggested v0.1.5 cut

If you want a tight v0.1.5 scope: **#1 (menu bar) + #2 (header
latency card) + #3 (Settings sidebar)**. All three are small, all
three are pure SwiftUI on top of the existing Rust/Swift bridge, and
together they materially improve daily usability without expanding
the engine surface area. Save **#4 (Connections pane)** for v0.1.6
once we have time to design the privileged-helper question, and
**#5 (Subscriptions)** for v0.2.0 when there's a use case from a
real user.
