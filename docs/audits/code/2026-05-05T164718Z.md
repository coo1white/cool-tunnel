## Critical / High narratives

### F5-1 — Required status checks not enabled on `main`

**Root violation.** The CI workflow at [`.github/workflows/ci.yml`](.github/workflows/ci.yml) runs the full polyglot lint floor: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`, `cargo deny check`, `swift-format lint --strict`, `shellcheck`, plus release-target builds for both `aarch64-apple-darwin` and `x86_64-apple-darwin`. This is the safety net that catches every other class of finding in this audit. But branch protection on `main` has *only* the PR-review requirement enforced (`required_approving_review_count: 1`); `required_status_checks` returns HTTP 404 — *not enabled*. A maintainer with merge rights can land a PR with red CI without GitHub objecting. The 1-reviewer rule provides social pressure, not a hard gate.

**Why it matters.** A project shipping a notarised macOS app, a Rust core that runs as a child process with elevated capability via subprocess piping, and a bundled `naive` binary verified by SHA256 has every reason to make CI a hard gate. The cost of an accidentally-merged regression on `main` is a notarised release that won't build, a CHANGELOG entry that has to retract, or — worst case — a security regression in `redaction.rs` that ships before anyone notices clippy was disabled.

**Remediation (single-direction).** Enable required status checks for the existing CI jobs. Single `gh api` PUT call:

```
gh api -X PUT repos/coo1white/cool-tunnel/branches/main/protection/required_status_checks \
  -f strict=true \
  -F 'contexts[]=Rust (build + clippy + test)' \
  -F 'contexts[]=Swift (format lint)' \
  -F 'contexts[]=ShellCheck'
```

Status-check contexts must match the `name:` keys in `ci.yml`. (Current names per inspection: `Rust (build + clippy + test)`, `Swift (format lint)`, `ShellCheck`.) `strict: true` requires the PR branch to be up to date with `main` before merge, which closes the "approved-against-stale-main" loophole.

This is one of three findings that would have been caught by an LTSC governance review of the repo settings rather than the code; F7-1 and F6-1 are the others.

## Cross-language patterns

Three patterns explain multiple findings or near-findings observed during the audit:

1. **Repo-settings hygiene lags code hygiene.** The codebase is in excellent shape — `#![forbid(unsafe_code)]`, `LazyLock` regex with `#[allow(clippy::expect_used)]` and explanatory doc comments, all five shell scripts open with `set -Eeuo pipefail`, governance docs (README, SECURITY, CONTRIBUTING, CHANGELOG) all present and substantive. But three findings (F5-1, F6-1, F7-1) are about *what GitHub knows*, not about *what the code does*. This pattern recurs in projects whose maintainer focus has been on code-quality discipline; the repo settings need their own review cadence.

2. **The lint floor is enforced in CI but not as a contributor onboarding step.** F1-1 (rustfmt drift) and F3-1 (cargo-deny missing) both come from the same root: the lint tools live in CI, not in a `make lint` / `cargo xtask lint` / pre-commit-hook surface that a contributor exercises locally before pushing. CI-only lint gates produce slow feedback cycles and let drift accumulate post-merge.

3. **Idempotency at the script edge.** F4-1 is a single-file violation, but it is symptomatic of one of the hardest invariants to keep across shell tooling: every script run on a converged system should produce zero diff. `fetch_naive.sh` updates the timestamp on every run regardless of content. A wider grep over `scripts/` for similar patterns is worth doing in a follow-up cycle.

## What is solid

Calibration check — these are not pad. Each is a specific control verified working.

1. **`#![forbid(unsafe_code)]` at [`core/src/lib.rs:1`](core/src/lib.rs:1)**, with the rationale stated in the `# Safety` rustdoc section: "There is no FFI layer; the Swift app communicates with this engine over a subprocess using newline-delimited JSON." This is the strongest possible Rust-safety posture — not just a soft default, an enforced absence — and the rationale links it to the Swift-side IPC discipline that justifies it.

2. **Regex `.expect()` callsites are the canonical safe pattern at [`core/src/redaction.rs:79–134`](core/src/redaction.rs:79).** Each of the four redaction regexes is a `static LazyLock<Regex>` with an `#[allow(clippy::expect_used)]` annotation and a doc comment naming why the regex is compile-time-known-infallible. Test code is wrapped with `#[allow(clippy::unwrap_used, clippy::expect_used)]` at the module level. The codebase's `.unwrap()` / `.expect()` outside tests is *exclusively* this pattern.

3. **All shell scripts open with `set -Eeuo pipefail`** ([`build_rust_core.sh:31`](scripts/build_rust_core.sh:31), [`cut_release.sh:45`](scripts/cut_release.sh:45), [`package_release.sh:23`](scripts/package_release.sh:23), [`fetch_naive.sh:26`](scripts/fetch_naive.sh:26), [`security_check.sh:31`](scripts/security_check.sh:31)) — and `fetch_naive.sh` uses the textbook `mktemp -d` + `trap 'rm -rf' EXIT` cleanup pattern at [lines 30–32](scripts/fetch_naive.sh:30). Zero `eval` across the script tree.

4. **The Rust↔Swift IPC contract is documented as a single source of truth on the Rust side** at [`core/src/protocol.rs:1–37`](core/src/protocol.rs:1) — frame examples for request, response, error, and event are in the module docs — with the Swift mirror at [`COOL-TUNNEL/Core/Protocol.swift:1`](COOL-TUNNEL/Core/Protocol.swift:1) explicitly cross-referencing it. A `protocol_roundtrip` integration test verifies lockstep (6 / 6 pass). This is the boundary discipline the polyglot prompt's B1 tenet expects, applied without fanfare.

5. **CI workflow [`.github/workflows/ci.yml`](.github/workflows/ci.yml) is comprehensive**: every linter from the polyglot crib sheet runs (`cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`, `cargo deny check`, `swift-format lint --strict`, `shellcheck`), plus release-target builds for both Apple Silicon and Intel. The CI invocations use `--locked` so no dependency drift can sneak past via a Cargo.lock regeneration. Combined with `dependabot.yml`, this is the polyglot G6 bar met. The remaining gap (F5-1) is making this CI mandatory for merge.
