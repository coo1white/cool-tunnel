## Critical / High narratives

### U2-1 тАФ Custom button styles have no visible focus state

The Maltese theme's `ModeChipStyle` and `SoftButtonStyle` (in
`MalteseTheme.swift:332` and `:391` respectively) define
`isActive` (running-and-matching), `configuration.isPressed`,
and disabled-via-tint. They do not define a focus state. SwiftUI
applies the system focus ring only when a button uses the
default style; a `ButtonStyle` that returns its own `body` opts
out of the ring entirely. The result: a keyboard-only user (or
a power user driving the app via the keyboard) can Tab through
the chips, Stop, Diag, Latency, Settings, the Add/Remove
buttons, and the Back/Done buttons тАФ and **the focused control
is not visibly distinguishable from any other**. Apple's HIG
treats this as a hard requirement, not polish.

**Remediation (single-direction)**: in both
`ModeChipStyle.makeBody` and `SoftButtonStyle.makeBody`, add
`.focusable()` already-implicit-via-Button, then `@FocusState`
read inside the style isn't possible (ButtonStyle has no
environment), so the cleaner fix is to add a SwiftUI
`.buttonStyle(.borderless)` fallback OR wrap the body in
`.focusable(true).focusEffect(.automatic)` (macOS 14+) which
draws the system focus ring back on. The smallest change is:
add `.contentShape(Capsule())` for hit-testing and use
`.focusEffectDisabled(false)` тАФ but the explicit fix is to
read `@Environment(\.isFocused)` (macOS 15+) inside the
style's body and apply a 2pt accent-coloured ring overlay
when focused. Single direction; no choice required.

### U5-2 тАФ Reduced-motion preference is unobserved

Cool Tunnel runs five distinct repeating or transition
animations: status-pill pulse (HeaderView:162), icon bounce on
state change (HeaderView:101 + ControlPanelView:127),
window-background cross-fade (ContentView:96-107),
empty-state sparkle pulse (LogConsoleView:113), and the
log-tail scroll animation (LogConsoleView:85). The codebase
does have an animation-gating system тАФ
`PerformanceProfile.repeatingSymbolEffectsAllowed` and
`.animationScale` тАФ but the gate is on **hardware tier**
(older Intel Macs get reduced animations), not on the user's
**preference** (System Settings тЖТ Accessibility тЖТ Display тЖТ
Reduce motion). A user who set Reduce motion in their OS
still sees every animation at full intensity in Cool Tunnel.
This is a contract violation: the OS-level setting exists
specifically so individual apps don't have to ask.

**Remediation (single-direction)**: extend
`PerformanceProfile` (or add a sibling `MotionPolicy` enum)
that `OR`s the existing hardware gate with `NSWorkspace.shared
.accessibilityDisplayShouldReduceMotion`. Every existing call
site that reads `PerformanceProfile.current
.repeatingSymbolEffectsAllowed` already has the right shape тАФ
the policy just becomes "false if hardware tier is `.light`
OR if the user requested reduced motion." Removes animations
under reduce-motion; leaves the rest untouched. Single
direction; no choice required.

### U6-1 тАФ Zero localizable strings

Every visible string in `Views/` is a `Text("тАж")` literal.
The project ships no `Localizable.strings`, no `.xcstrings`
catalog, no `String(localized:)` calls. The product's primary
use case (per the README and the project's domain registry
hint at `coo1white`) is Chinese-mainland users routing
around the GFW; shipping an English-only UI to that
audience is a real friction cost today, not a future-work
item. Beyond the language barrier: the pluralisation bug at
LogConsoleView:45 ("1 lines") and the LTR-encoded chevron at
SettingsView:87 are both downstream consequences of the
same root тАФ strings and direction-encoded glyphs are written
once in English and never run through a locale-aware layer.

**Remediation (single-direction)**: this is the one finding
in the audit where a single-direction remediation does not
exist. The choice is between (A) adopt the macOS-native
`String Catalogs` (.xcstrings) introduced in Xcode 15 and
add a zh-CN translation pass, or (B) acknowledge in the
README that the UI is English-only by design. Both are
defensible; one is a real engineering project (string
extraction + translation infrastructure + pluralisation
fixes), the other is a single doc paragraph. **DEFER** тАФ
operator picks the direction. This audit's job is to flag
that the gap exists and quantify it (30+ literal Text
constructions), not to choose the resolution.

## Strategic batching

Findings group into 4 ship batches. Tenet-mixed by surface, not
single-tenet sweeps:

### Batch A тАФ HeaderView polish (3 findings)

- U1-4 (errorBanner contrast)
- U3-2 (errorBanner truncates multi-line messages)
- U4-2 (status pill not combined AX element)

All three live in `HeaderView.swift` and touch the same two
sub-views (errorBanner, statusPill). Single PR; one rebuild.

### Batch B тАФ Design-system pinning (5 findings)

- U1-1 (mixed corner radii)
- U1-2 (mixed border widths)
- U1-3 (inline font literal in ConnectionFormView)
- U2-3 (three views forking on `\.colorScheme`)
- U5-1 (animation duration/easing pairs ad-hoc)

All five trace back to "the design system is named but not
enforced." Resolution: pin a radius scale (4 / 6 / 8 тАФ pick
two), pin a border width (one value, e.g. 0.6), pin an
animation token enum (`Motion.fast` / `.standard` / `.slow`
or named springs), and migrate the three colorScheme readers
into `CTPalette` adaptive tokens. Single PR; touches
`MalteseTheme.swift` plus minor view edits.

### Batch C тАФ Accessibility contract (4 findings)

- U2-1 (custom button styles missing focus-visible) **High**
- U4-1 (decorative SF Symbols not hidden)
- U4-3 (mode chip uses Button+Value instead of selectable
  trait)
- U5-2 (reduced-motion preference unobserved) **High**

All four are AX-contract violations. Same PR scope: extend
`PerformanceProfile` to honour reduce-motion, add focus rings
to the two custom button styles, mark decorative SF Symbols
across `Views/`, and tighten the mode chip's AX traits.

### Batch D тАФ i18n decision (2 findings)

- U6-1 (zero localizable strings) **High**
- U3-1 (broken pluralisation, downstream of U6-1)
- U6-2 (chevron.left vs chevron.backward, direction-encoded)

Conditional batch тАФ depends on the decision in U6-1's narrative
(adopt String Catalogs vs. acknowledge English-only-by-design).
If A is chosen, Batch D becomes a multi-week project; if B is
chosen, Batch D is a one-line README edit + the chevron swap.
**DO NOT** ship A and B's fix variants on the same PR.
